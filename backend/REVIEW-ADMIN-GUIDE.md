# Document Review System — Admin Guide

This guide walks you through the complete process of setting up and running a document review cycle. Follow these steps in order for each new review.

---

## Overview

The review system lets a small committee vote on each section of a governing document (CC&Rs, Bylaws, or Rules of Conduct). Two data files are prepared from Word documents:

1. **Seed data** (from the Field Guide) — The annotated guide with classifications, "Why it's here", and "What you can do" for each section. This is what reviewers see and vote on.
2. **Document text** (from the Governing Document) — The actual legal text. Displayed when reviewers click "View Text" on a section.

Both files are generated by a single parse script, uploaded to S3, and then loaded into a review cycle through the Admin Console in one step.

### Complete Workflow

| Step | What | Where |
|------|------|-------|
| 1 | Parse both documents and upload to S3 | Terminal |
| 2 | Create cycle and load data | Admin Console |
| 3 | Add reviewers to Cognito group | Terminal (as needed) |
| 4 | Notify reviewers | Email/Slack/etc. |
| 5 | Monitor progress | Review Dashboard |
| 6 | View aggregate results | Review Dashboard |
| 7 | Record final decisions | Admin Console — Summary Ballot |
| 8 | Start next cycle (optional) | Admin Console |

## Prerequisites

Before starting, ensure you have:

- [ ] **Admin access** — You must be in the `review-admins` Cognito group
- [ ] **pandoc** installed — `brew install pandoc` (used to convert Word docs)

- [ ] **AWS CLI** configured — Required for S3 uploads (`aws configure`)
- [ ] **Field Guide** Word document (.docx) — The annotated guide for the document being reviewed
- [ ] **Governing Document** Word document (.docx) — The actual legal document (CC&Rs, Bylaws, etc.)

### Important file locations

| Item | Location |

| Output seed files (JSON) | `backend/seeds/` |

## Step 1: Parse Both Documents and Upload to S3

A single script parses both the Field Guide and the Governing Document together. It validates that the articles and sections align, generates a shared `documentSetId` for integrity checking, and produces two matched output files.

### 1a. Run the unified parse script

From the project root directory:

```bash
node backend/scripts/parse-review-docs.js \
  --field-guide "path/to/MMPOA_CCRs_Field_Guide_V1.docx" \
  --governing-doc "path/to/MMPOA_CCRs_V1.docx" \
  --output-dir backend/seeds \
  --name ccrs-2026-01 \
  --upload
```

**Arguments:**

| Argument | Description | Example |
|----------|-------------|---------|
| `--field-guide` | Path to the Field Guide .docx file | `"documents/Field_Guide.docx"` |
| `--governing-doc` | Path to the Governing Document .docx file | `"documents/CCRs.docx"` |
| `--output-dir` | Directory for output JSON files | `backend/seeds` |
| `--name` | Base name for output files | `ccrs-2026-01` |
| `--upload` | (Optional) Upload both files to S3 automatically | |

### 1b. What it produces

The script creates two matched files:

| File | Contains | Used For |
|------|----------|----------|
| `ccrs-2026-01-seed.json` | Field Guide annotations (classifications, guidance) | Reviewer voting content |
| `ccrs-2026-01-text.json` | Governing document legal text | "View Text" popup |

Both files contain a shared `documentSetId` (e.g., `ccrs-2026-01-a1b2c3d4e5f6`). The Admin Console validates this ID when loading to ensure the files are matched.

### 1c. Review the output

The script prints a summary to the terminal showing:
- Article counts and section counts from both documents
- Validation results (errors if articles don't match, warnings for minor differences)
- Classification breakdown (required_by_law, best_practice, community_choice, etc.)
- The generated documentSetId

### 1d. Verify S3 upload

If you used `--upload`, verify the files are in S3:

```bash
aws s3 ls s3://mmpoa-review-seeds/
```

You should see both `ccrs-2026-01-seed.json` and `ccrs-2026-01-text.json`.

### Naming convention for files

Use the format `{document}-{year}-{sequence}`:
- `ccrs-2026-01` — first CC&Rs review in 2026
- `ccrs-2026-02` — second round (if needed)
- `bylaws-2026-01` — first Bylaws review
- `conduct-2026-01` — first Rules of Conduct review

## Step 2: Create Cycle and Load Data

The Admin Console combines cycle creation and data loading into a single step.

1. Log in to the website and go to the **Board Portal**
2. Scroll to the bottom and click the **Review Admin Console** card
3. On the **Create & Load Cycle** tab, fill in:

| Field | Description | Example |
|-------|-------------|---------|
| **Document** | Which governing document (dropdown) | CC&Rs, Bylaws, or Rules of Conduct |
| **Cycle ID** | Unique identifier for this review round | `CCRS-2026-01` |
| **Title** | Human-readable title shown to reviewers | `CC&Rs Review — April 2026` |
| **Copy from prior cycle** | Optional — prior cycle to carry forward unresolved sections (dropdown) | Select from existing cycles |
| **Document Files to Load** | Matched file pair from S3 (dropdown) | `ccrs-2026-01` |

4. Click **Create Cycle & Load Data**

The system performs three operations in sequence:
1. Creates the review cycle
2. Seeds the review content from the seed file
3. Loads the document text from the text file
4. Validates the documentSetId match between both files

You'll see progress messages (Step 1/3, 2/3, 3/3) and a final success message with section counts and documentSetId verification.

### The Existing Cycles panel

At the top of every tab, you can see all existing cycles in a table showing Cycle ID, Document, Title, Status, and Created date. Click **Refresh** to update the list. Cycle dropdowns throughout the Admin Console are automatically populated from this list.

### Cycle ID naming convention

Use the format `DOCUMENT-YEAR-SEQUENCE`:
- `CCRS-2026-01` — first CC&Rs review in 2026
- `CCRS-2026-02` — second round (carrying forward unresolved items)
- `BYLAWS-2026-01` — first Bylaws review in 2026
- `CONDUCT-2026-01` — first Rules of Conduct review in 2026

### Carrying forward from a prior cycle

When you select a prior cycle in the "Copy from" dropdown, the system will:
- Look at the Summary Ballot decisions from the prior cycle
- Copy forward only sections that were **NOT Approved** and **NOT Removed**
- Sections marked Disapprove or with no decision carry into the new cycle
- This lets the committee focus on unresolved items in subsequent rounds

## Step 3: Add Reviewers

Each reviewer needs to be added to the `reviewers` Cognito group. Admins need to be in the `review-admins` group.

### Add a reviewer

```bash
aws cognito-idp admin-add-user-to-group \
  --user-pool-id us-east-1_tfk0ub8lC \
  --username user@example.com \
  --group-name reviewers \
  --region us-east-1
```

### Add an admin

```bash
aws cognito-idp admin-add-user-to-group \
  --user-pool-id us-east-1_tfk0ub8lC \
  --username user@example.com \
  --group-name review-admins \
  --region us-east-1
```

### List group members

```bash
aws cognito-idp list-users-in-group \
  --user-pool-id us-east-1_tfk0ub8lC \
  --group-name reviewers \
  --region us-east-1 \
  --query 'Users[*].Username' --output table
```

**Important:** Users must sign out and sign back in for group changes to take effect.

## Step 4: Notify Reviewers

Tell reviewers:

- Log in at **mmpoaii.org**
- Go to the **Board Portal** or **Homeowner Portal**
- Scroll to the bottom and click **Review Documents**
- Select the review cycle
- For each article, vote on every section (Approve / Disapprove / Discuss)
- Click "View Text" to see the original legal language
- Votes and notes save automatically
- When all sections are voted, click **Submit Ballot** on the articles page

---

## Step 5: Monitor Progress

As an admin, go to the Review Dashboard (click the reviewer card, not admin):
- Select the cycle to see the article list
- Each article shows colored badges: Approved (green), Disapproved (red), Discuss (yellow), Waiting (grey)
- Overall progress bar and totals are shown at the top
- **Note:** You see only your own progress until ballots are submitted

---

## Step 6: View Aggregate Results

After reviewers submit their ballots:
1. Go to the article list for the cycle
2. Click **View Aggregate Results** (appears after you submit your own ballot)
3. The aggregate view shows:
   - Vote counts per section (Approve / Disapprove / Discuss)
   - All reviewer notes
   - Unanimous approvals are highlighted

**Important:** Reviewers must submit their ballot before viewing aggregates. This prevents vote influence.

---

## Step 7: Record Final Decisions

1. Go to the **Admin Console** — **Summary Ballot** tab
2. Select the Cycle from the dropdown and click **Load Summary**
3. For each section, you'll see aggregate votes and notes
4. Click a decision button for each section — it saves immediately

| Decision | Meaning | Carries Forward? |
|----------|---------|-----------------|
| **Approve** | Section accepted as-is | No |
| **Disapprove** | Section needs revision | Yes |
| **Remove** | Section should be deleted | No |

---

## Step 8: Start Next Cycle (if needed)

If some sections were marked Disapprove or left undecided:

1. Go to the **Create & Load Cycle** tab
2. Enter a new Cycle ID (e.g., `CCRS-2026-02`)
3. In **Copy from prior cycle**, select the previous cycle (e.g., `CCRS-2026-01`)
4. Select the same file pair (or a new one if documents were updated)
5. Click **Create Cycle & Load Data**
6. Only unresolved sections carry forward from the prior cycle

## Troubleshooting

**"pandoc: command not found"**
- Install pandoc: `brew install pandoc`

**Parse script shows 0 articles or wrong count**
- Verify the Word documents follow the expected format (see Word Document Format section below)
- Try running pandoc manually to inspect the markdown: `pandoc input.docx -t markdown -o debug.md`
- Check for missing classification labels or "Why it's here" fields

**Parse script reports validation errors**
- If an article exists in the Field Guide but not the Governing Document, the script will error
- Check that both documents cover the same articles
- Minor section count differences are reported as warnings, not errors

**"No file pairs found in S3" in Admin Console dropdown**
- The parse script must be run with `--upload` first
- Files must follow the naming convention: `{name}-seed.json` and `{name}-text.json`
- Both the seed and text files must be present for a pair to appear
- Check S3: `aws s3 ls s3://mmpoa-review-seeds/`

**"Failed to fetch" in Admin Console**
- Sign out and sign back in to refresh your authentication token
- Check that you are in the `review-admins` Cognito group

**Reviewer can't see the Review Documents section**
- They must be in the `reviewers` Cognito group
- They must sign out and sign back in after being added to the group

**"Cycle already exists" error**
- Each Cycle ID must be unique. Use a different ID (increment the sequence number)

**"Document text not available for this section"**
- The document text was not loaded for this cycle
- Re-run Create & Load Cycle, or check that the text file was in S3 when the cycle was created

**Votes not showing in article list or aggregate**
- This was a known bug (URL encoding issue with `#` in section IDs) that has been fixed
- If you see stale data, votes may need to be re-cast on affected sections

**"Document Set IDs do not match" warning**
- The seed and text files were generated from different document sets
- Re-run the parse script with both documents together to generate matched files

**S3 upload fails with "access denied"**
- Ensure your AWS CLI is configured with credentials that have access to the `mmpoa-review-seeds` bucket
- Run `aws sts get-caller-identity` to verify your credentials

## Word Document Formats

### Field Guide Format

The parse script expects this structure in the Field Guide .docx file:

**Article headings** (Roman numerals):
```
Article I — Definitions
Article II — Property Rights in Common Areas
```

**Section cards** (within each article):
```
Art. I, S1 — Association
RECOMMENDED BEST PRACTICE

Why it's here: [explanation text]

What you can do: [guidance text]

Community Impact: [optional — only for some sections]
```

**Classification labels** — exactly one per section:
- `REQUIRED BY TEXAS LAW`
- `REQUIRED BY CITY OF AUSTIN`
- `RECOMMENDED BEST PRACTICE`
- `COMMUNITY CHOICE`

### Governing Document Format

**Article headings**:
```
ARTICLE I - DEFINITIONS
```

**Section headings**:
```
Section 1. "Association."
```

The full text below each section heading is captured as the legal text.

## Architecture Notes

### Data stored in DynamoDB (`mmpoa-reviews` table)

| Data | PK | SK Pattern |
|------|-----|------------|
| Cycle metadata | `CYCLE#{cycleId}` | `META` |
| Section content (Field Guide) | `CYCLE#{cycleId}` | `CONTENT#ART-{nn}#SEC-{nn}` |
| Document text (legal) | `CYCLE#{cycleId}` | `DOCTEXT#ART-{nn}#SEC-{nn}` |
| Reviewer votes | `CYCLE#{cycleId}` | `VOTE#ART-{nn}#SEC-{nn}#USER#{sub}` |
| Ballot status | `CYCLE#{cycleId}` | `BALLOT#USER#{sub}` |
| Admin decisions | `CYCLE#{cycleId}` | `SUMMARY#ART-{nn}#SEC-{nn}` |

### Data flow

```
Field Guide (.docx)  +  Governing Document (.docx)
                  \      /
            parse-review-docs.js
                  /      \
     {name}-seed.json    {name}-text.json
          (shared documentSetId)
                  \      /
                Upload to S3
                     |
          Admin Console: Create & Load Cycle
             /                    \
     cycle-seed Lambda      doctext-save Lambda
             |                      |
     CONTENT# in DynamoDB    DOCTEXT# in DynamoDB
             |                      |
     Reviewer sees annotations    "View Text" popup
     and votes on each section    shows legal language
```
